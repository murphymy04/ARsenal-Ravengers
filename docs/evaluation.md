# Evaluation

This is the consolidated record of how we measure ArgusEye's two hard sub-problems — face recognition and speaker classification — and what we know about their failure modes.

The eval harness lives in [ar-glasses/eval/](../ar-glasses/eval/) and the source notes that fed this doc are in [ar-glasses/eval/EVAL_NOTES.md](../ar-glasses/eval/EVAL_NOTES.md).

## 1. Face recognition

Recognition is the easier of the two problems for our use case (1-on-1, near-frontal, conversational distance) and is not currently the bottleneck.

### Setup

- Embedding model: EdgeFace `edgeface_base`, 512-d.
- Match decision: cosine similarity ≥ `MATCH_THRESHOLD = 0.4` against the gallery of stored embeddings ([processing/config.py](../ar-glasses/processing/config.py)).
- ROC across the LFW pairs benchmark prepared by [eval/prepare_lfw.py](../ar-glasses/eval/prepare_lfw.py) and run via [eval/evaluate.py](../ar-glasses/eval/evaluate.py).
- Curve: `ar-glasses/eval_roc.png`.

### Results

The threshold of 0.4 was picked from the ROC's knee — well above the noise floor for unrelated faces (~0.1 cosine sim) and below the typical same-person score (~0.5–0.7 in our test footage).

Failure modes we've observed:

- **Profile views** during head turns drop similarity below threshold for a few frames. Temporal smoothing (`TEMPORAL_SMOOTHING_FRAMES = 7`) absorbs these; without it, recognition would flicker.
- **Heavy compression artifacts** from the Inmo Air 3's H.264 stream cause occasional false negatives. We mitigate with `FACE_BLUR_THRESHOLD = 70.0` (Laplacian variance) — frames sharper than that are accepted for embedding even if compression is visible.
- **Lookalikes** (siblings, close family) — not measured. The 0.4 threshold is forgiving and would likely cause merges. Out of scope for the prototype.

Recognition latency in the pipeline:

| Operation | Latency | Backend |
|---|---|---|
| Face detection | 5–10 ms | OpenCV Res10-SSD |
| Face embedding (per face) | 10–20 ms | EdgeFace |
| Gallery match (cosine over ≤20 emb × N people) | < 1 ms | NumPy |
| Temporal smoothing & tracking | < 1 ms | bbox proximity |

At 30 fps with one or two faces in frame, total per-frame vision cost is ~25–35 ms — within real-time budget. `VISION_STRIDE = 5` gives ~5× headroom on offline videos.

## 2. Speaker classification

This is the **dominant accuracy bottleneck** in the system. Robust 2-speaker diarization in conversational audio with a wearer-mic and a room mic is itself an open research area.

### Setup

- Ground truth: [eval/transcripts_to_annotate.txt](../ar-glasses/eval/transcripts_to_annotate.txt) — 98 hand-annotated segments across three test videos: `timur_myles_2`, `timur_peter`, `timur_will`. Generated by running `python -m pipeline.live` on each video, extracting segments, then hand-correcting speaker labels.
- Eval script: [testing/eval_speaker_accuracy.py](../ar-glasses/testing/eval_speaker_accuracy.py). Runs the adaptive VAD + RMS pipeline on each video and compares frame-level classifications to ground truth via per-segment majority vote.
- Production config: `seed=0.035`, `alpha=0.3`, `VAD_THRESHOLD = 0.35` ([processing/config.py](../ar-glasses/processing/config.py), [processing/vad_speaker.py](../ar-glasses/processing/vad_speaker.py)).

### Strategies tried

| Approach | Best segment error | Notes |
|---|---|---|
| Baseline adaptive (seed=0.035, alpha=0.3) | 34.7% | Current production config. |
| Parameter sweep (seed/alpha/multipliers) | 30.6% | Best: seed=0.02, alpha=0.1, h=1.5, l=0.3. |
| Temporal smoothing (sliding majority vote) | 28.6% | Window=5 frames post-classification. |
| Biased boundary (closer to mean_low) | 27.6% | bias=0.45 instead of 0.5 midpoint. |
| Speaker inertia (require N frames to switch) | 27.6% | 8-frame inertia. |
| Smoothed-RMS adaptive | **25.5%** | 5-frame RMS smoothing window. |
| Segment-level adaptive | 25.5% | Mean RMS per segment vs boundary. |
| Smoothed RMS + segment reclassify | 25.5% | Two-stage: frame adaptive then segment reclassify. |
| Spectral features (low/high freq energy ratio) | 34.7% | Proximity-effect feature; didn't help standalone. |
| Combined RMS + spectral | 27.6% | Weighted normalized score; no better than RMS alone. |
| Two-pass adaptive | 34.7% | Run twice, second pass seeded from first. |
| Batch k-means (offline oracle) | 34.7% | k=2 on all speech RMS. |
| Percentile-seeded | 33.7% | Seed from median of first 50 speech frames. |

### The hard ceiling

Even with the **oracle-optimal static boundary per video** (perfect hindsight, best possible single threshold):

| Video | Optimal boundary | Segment error | Frame accuracy |
|---|---|---|---|
| `timur_myles_2` | 0.008 | 3 / 22 = 13.6% | 79.8% |
| `timur_peter`    | 0.044 | 7 / 31 = 22.6% | 78.0% |
| `timur_will`     | 0.096 | 14 / 45 = 31.1% | 70.1% |
| **Total** | — | **24 / 98 = 24.5%** | — |

A logistic regression trained on **all** features we have (RMS, spectral ratio, low/high energy, VAD prob), even overfit to training data, only reaches 71% frame accuracy and 29.6% segment error. **Our features are insufficient.**

### Why it's hard

Per-speaker RMS distributions overlap heavily at the frame level, especially in noisy recordings:

| Video | Wearer mean | Other mean | Gap | Overlap |
|---|---|---|---|---|
| `timur_myles_2` | 0.043 | 0.015 | 0.028 | moderate |
| `timur_peter`    | 0.099 | 0.047 | 0.052 | high |
| `timur_will`     | 0.106 | 0.072 | 0.034 | very high |

No single-threshold RMS method can do better than ~25% error on this corpus. The ceiling is the data, not the algorithm.

### What would actually move the needle

To get below 10% error we need a fundamentally different signal:

- **Speaker embeddings** (resemblyzer, pyannote, speechbrain ECAPA-TDNN). Trades CPU for accuracy.
- **Better mic hardware**: a near-field bone-conducted wearer mic plus active noise suppression on the room mic. The cleanest single intervention.
- **Visual channel**: face detection is already in the pipeline. MediaPipe's `jawOpen` blendshape (already wired up via `SPEAKING_BACKEND = "mediapipe"`) is one knob — when the wearer's face is not visible and the other person's `jawOpen` is high, attribution to "other" becomes near-certain regardless of RMS.

These are non-trivial and out of scope for the current prototype, hence the explicit "1-on-1 only" caveat in [docs/requirements.md](requirements.md).

## 3. End-to-end latency

| Stage | Typical latency |
|---|---|
| Audio window (`LIVE_BUFFER_SECONDS`) | 7 s (configurable) |
| Whisper transcription (Groq) | < 1 s for a 7 s clip |
| Speaker assignment + segment merge | < 100 ms |
| Graphiti `add_episode` (background) | 5–30 s for a normal-length conversation |
| Recognition → HUD WebSocket send | < 200 ms after the cooldown gate |
| Cooldown gate per person | `RETRIEVAL_COOLDOWN_SECONDS` (default 10 s) |

Saves to the knowledge graph are fully async, so they never block the dashboard or the HUD path. The wearer-perceived latency for "see a face → get a context card" is dominated by the cooldown gate, not the search.

## 4. Reproducing these numbers

```bash
cd ar-glasses

# Speaker classification eval (writes per-strategy results)
python testing/eval_speaker_accuracy.py

# Face recognition ROC
python eval/prepare_lfw.py    # one-time, downloads + prepares LFW pairs
python eval/evaluate.py       # writes ../eval_roc.png
```

If you're adding a new strategy, slot it into the table in `eval_speaker_accuracy.py` and update the table in this doc. The ground-truth annotations in `eval/transcripts_to_annotate.txt` are the only thing that should not be regenerated casually — if you re-extract segments you must re-annotate, and the numbers above stop being comparable.
